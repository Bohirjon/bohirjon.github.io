Dependency Inversion Principle (DIP)

❌ Tight Coupling
// Depends on concrete implementation, not abstraction
public class OrderService
{
    private readonly CreditCardPayment _payment;

    public OrderService()
    {
        _payment = new CreditCardPayment(); // tightly coupled
    }

    public void ProcessOrder(decimal amount)
    {
        _payment.Charge(amount); // depends on implementation
    }
}

🚨 Problems with this approach:

🔗 Tight Coupling: OrderService directly depends on CreditCardPayment implementation
🚫 Hard to swap: Cannot easily add new payment methods
🧪 Hard to Test: Cannot mock payment method for unit testing
💥 Breaks easily: Changes in payment logic break OrderService
