1. Implement Access Control

Description
Threats
Implementation
Tools

Definition: Authentication and authorization mechanisms that systematically enforce access constraints on principals (users/services) attempting to access resources

Fundamental axiom: Authentication (identity verification) ≠ Authorization (privilege verification)

  • The Risks of Loose Access Control Policies: An attacker could take advantage of a loosely configured access control policy to access data the organization did not intend to make accessible.
  • Exploiting Weak Links: Vulnerabilities in Application Access Control Components: An attacker could discover multiple access control components within an application and exploit the weakest.
  • Old Accounts Left Active: How Attackers Can Use Them to Steal Data: An administrator could forget to decommission an old account, and an attacker could discover that account and use it to access data
  • How Attackers Exploit Missing Default Deny Policies:An attacker could gain access to data that had a policy that dropped into a final step of allowing access. (Lack of default deny)
  • Force every request to though ACC :Ensure all requests are forced to go though access control verification
  • Centralized Authentication & Authorization:Centralizing access control avoids inconsistent implementations and lets you focus security efforts on reviewing and maintaining a single, reusable function across your codebase.
  • Deny-by-Default Posture: Implement allowlist-based security controls with explicit permission declarations.

OWASP ZAP with Access Control Testing & Active Scanner extensions

Open Policy Agent (OPA) for policy-as-code authorization

Keycloak/Auth0 for identity federation and centralized policy enforcement

OWASP ModSecurity Core Rule Set (CRS) for WAF-level access control

3. Validate All Inputs & Handle Exceptions

Description
Threats
Implementation
Vulnerabilities

Definition: Ensuring only properly formatted data enters a software system component

Key Concepts: Syntactic validity (correct form) and semantic validity (acceptable range)

  • SQL Injection: Manipulating database queries to access unauthorized data
  • Cross-Site Scripting (XSS): Injecting malicious scripts executed in users' browsers
  • Remote Command Execution: Injecting system commands to execute arbitrary code
  • Allowlisting vs Denylisting: Prefer allowlisting (known good) over denylisting (known bad)
  • Server-Side Validation: Never rely on client-side validation alone for security
  • Parametrized Queries: Use prepared statements/ORMs to prevent SQL injection
  • Encode user inputs: Always encode user input to prevent vulnerabilities and data corruption. <, >, & to &lt;, &gt;, &amp;

4. Address Security from the Start

Description
Threats
Implementation
Vulnerabilities
Tools

Definition: Design and implement security measures from the beginning of the application development lifecycle

Key Principle: Complexity is the enemy of security - design for clarity and transparency

  • Security by Obscurity: If protection relies only on code obfuscation, attackers can reverse-engineer and gain full access
  • Authorization Complexity: Complex designs lead to misconfiguration and insecure implementation by developers
  • Default Insecurity: Systems requiring manual security configuration often result in exposed components
  • Excessive Attack Surface: Unnecessarily exposed components provide additional attack vectors
  • Keep It Simple (KISS): Design systems that are easy to understand and reason about security-wise
  • Security by Design & Default: Make the path of least resistance result in a secure system
  • Minimize exposed components Identify and reduce exposed components - attackers cannot attack what's not there
  • Use Secure Architecture Patterns: Apply standard solutions reviewed and hardened against known threats

OWASP Top 10 2021-A04: Insecure Design

CWE-637: Unnecessary Complexity in Protection Mechanism

CWE-656: Reliance on Security through Obscurity

5. Secure By Default Configurations

Description
Threats
Implementation
Vulnerabilities
Tools

Definition: Products resilient against exploitation out of the box without requiring extensive configuration

Key Benefit: Reduces burden on developers by providing a secure starting point rather than requiring manual hardening

  • Default Credential Exploitation: Attackers gain unauthorized access using unchanged default or well-known credentials(admin,admin, 1234 brand-name)
  • Overly Permissive Settings: Attackers leverage excessive permissions to access sensitive resources
  • Unnecessary Features:Swagger, logs, stack-trace
  • Missing Security Headers: XSS and other attacks succeed due to lenient default security configurations 
    
// Example: Setting secure HTTP headers in ASP.NET Core
context.Response.Headers.Add("Content-Security-Policy", "default-src 'self'; script-src 'self';");
context.Response.Headers.Add("X-Content-Type-Options", "nosniff");
context.Response.Headers.Add("X-Frame-Options", "DENY");
context.Response.Headers.Add("Referrer-Policy", "no-referrer");
  • Infrastructure-as-Code (IaC): Deploy using declarative templates for automated provisioning with secure defaults
  • Least Privilege Configurations: Ensure cloud storage and resources are private by default with minimal access
  • Deny-by-Default Access: Implement allowlist-based access control instead of permissive defaults
  • Disable Unnecessary Features: Remove demo capabilities, accounts, and debug features in production

6. Keep your Components & libraries Secure

Description
Threats
Implementation
Vulnerabilities
Tools

Definition: Using secure libraries and frameworks while ensuring third-party components remain updated and free from vulnerabilities

Key Benefit: Prevents security-related design and implementation flaws without requiring extensive in-house security expertise

  • Outdated Dependencies: Attackers exploit known vulnerabilities in unpatched third-party components to gain unauthorized access
  • Malicious code in libraries: Compromised libraries or frameworks insert malicious code during the development process
  • Expose sensitive information : Third-party components without proper secure expose sensitive information
  • Dependency-Based DoS: Known vulnerabilities in external libraries are targeted to disrupt service availability
  • Select Trusted Sources: Download libraries from official repositories over secure links and prefer signed packages
  • Evaluate Library Quality: Consider popularity, activity, maturity, and complexity when selecting components
  • Continuous Vulnerability Monitoring: Implement automated scanning for known vulnerabilities in dependencies
    • OWASP Dependency-Check, Snyk
    • dotnet list package --vulnerable
  • Shift-Left Security: Integrate Software Composition Analysis (SCA) early in the development lifecycle
  • Regular Updates: Proactively update libraries and frameworks throughout the product lifecycle

7. Secure Digital Identities

Description
Threats
Implementation
Vulnerabilities
Tools

Definition: Digital Identity is a unique representation of an individual or entity engaging in an online transaction

Key Components: Authentication (verification of identity claims) and session management (maintaining authentication state)

  • Brute Force Attacks: Attackers repeatedly guess passwords to gain unauthorized access
  • Stolen Credential: Attackers use compromised username/password pairs from other breaches to gain access
  • Weak Passwords: Users choose easily guessable passwords, making accounts vulnerable to brute-force attacks
  • Authentication Assurance Levels:
    • Level 1: Single-factor authentication (e.g., username/password)
    • Level 2: Two-factor authentication (e.g., password + SMS code)
    • Level 3: Multi-factor authentication with hardware tokens, this is typically done through hardware cryptographic modules
    • Strong Password Policies:
      • Minimum length (e.g., 12 characters)
      • Complexity requirements (uppercase, lowercase, numbers, symbols)
      • Don't allow easy or common passwords
      • JWT Best Practices:
        • Use RS256 or ES256 instead of HS256 for better security
        • Set appropriate expiration times (e.g., 15 minutes for access tokens, 7 days for refresh tokens)
        • ALWAYS VALIDATE signature

8. Leverage Browser Security Features

Description
Threats
Implementation
Vulnerabilities
Tools

Browser Gateway Security: Browsers are the gateway to the web for most users. It's critical to employ robust security measures to protect users from various threats.

Opportunistic Security: Web applications cannot verify that browsers follow security guidance, so these measures should be seen as additional hardening layers that complicate an attacker's efforts.

Browser Support: Security features differ between browsers and versions. Explicitly setting security headers unifies browser behaviors and reduces maintenance effort.

  • Cross-Site Scripting (XSS) Attacks: Inadequate Content Security Policy settings can allow attackers to inject malicious scripts into web pages.
  • Clickjacking Attacks: Missing X-Frame-Options headers enable attackers to trick users into unintended interactions with disguised web elements.
  • Information Leakage: Improper Referrer-Policy settings can expose sensitive data or user activities through HTTP headers.
  • MIME Type Confusion: Without X-Content-Type-Options headers, attackers can exploit vulnerabilities by disguising malicious scripts as benign file types.
  • Session Hijacking: Insecure cookie settings can allow attackers to gain unauthorized access to user accounts.
  • Cross-Origin Resource Sharing (CORS) Issues: Misconfigured CORS can grant unauthorized access to resources, compromising data confidentiality and integrity.
  • Prevent Information Disclosure:
    • HTTP Strict Transport Security (HSTS): Ensure HTTPS connections only
    • Content Security Policy (CSP): Auto-upgrade HTTP to HTTPS
    • Referrer-Policy: Control information transmitted between websites
    • Secure Cookie Flag: Prevent cookies from being sent over unencrypted HTTP
  • Mitigate XSS Impact:
    • Content Security Policy (CSP): Disable inline JavaScript/CSS
    • Host Allowlist CSP: Block third-party JavaScript to reduce attack surface
    • Strict CSP with nonces/hashes: Robust protection using script-src directive
    • Trusted Types: Prevent DOM-based XSS by ensuring only secure data types are inserted into the DOM
    • HttpOnly Cookie Flag: Prevent JavaScript from accessing cookies
  • Prevent Clickjacking:
    • X-Frame-Options (XFO): Prevent content embedding in other sites
    • CSP frame-* directives: Fine-grained control of site embedding
  • Control Advanced Capabilities:
    • Permission Policy: Restrict access to browser hardware (microphones, cameras, etc.)
  • Prevent CSRF Attacks:
    • Same-Origin Cookies: Mark cookies as SameSite to mitigate cross-origin information leakage
    • Fetch Metadata Request Headers: Implement Resource Isolation Policy to protect against CSRF
