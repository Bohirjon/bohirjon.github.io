1. Implement Access Control

Description
Threats
Implementation
Tools

Definition: Authentication and authorization mechanisms that systematically enforce access constraints on principals (users/services) attempting to access resources

Fundamental axiom: Authentication (identity verification) ≠ Authorization (privilege verification)

  • The Risks of Loose Access Control Policies: An attacker could take advantage of a loosely configured access control policy to access data the organization did not intend to make accessible.
  • Exploiting Weak Links: Vulnerabilities in Application Access Control Components: An attacker could discover multiple access control components within an application and exploit the weakest.
  • Old Accounts Left Active: How Attackers Can Use Them to Steal Data: An administrator could forget to decommission an old account, and an attacker could discover that account and use it to access data
  • How Attackers Exploit Missing Default Deny Policies:An attacker could gain access to data that had a policy that dropped into a final step of allowing access. (Lack of default deny)
  • Frorce every request to though ACC :Ensure all requests are forced to go though access control verification
  • Centralized Authentication & Authorization:Centralizing access control avoids inconsistent implementations and lets you focus security efforts on reviewing and maintaining a single, reusable function across your codebase.
  • Deny-by-Default Posture: Implement allowlist-based security controls with explicit permission declarations.

OWASP ZAP with Access Control Testing & Active Scanner extensions

Open Policy Agent (OPA) for policy-as-code authorization

Keycloak/Auth0 for identity federation and centralized policy enforcement

OWASP ModSecurity Core Rule Set (CRS) for WAF-level access control

2. Use Cryptography to Protect Data

Description
Threats
Implementation
Vulnerabilities

Keep secrets safe: Passwords, personal info, payment details

First step: Sort your data by how private it is

  • Outdated encryption alghoritmsAn attacker could exploit weak or outdated encryption algorithms to decrypt sensitive information.
  • Sql Injection to steal data:Attackers can steal sensitive data using sql injection
  • improperly stored keys:Improperly stored encryption keys could be compromised, leading to unauthorized data access.
  • No dot store passwords in plain text:No dot store plain text in database, encrypt with random generated salt
  • Keep Secrets in Vaults: Use special tools to store app secrets.
  • Use HTTPS, TLS 1.2 or 1.3:Disable TLS versions older than 1.2.
  • Trusted certificate authoirities:Make sure you're using certificate from trusted certificate authorities.
  • Change Keys Regularly: Update your keys and plan for when they're stolen.
  • Use secure flag:When setting cookies, always utilize the “secure” flag to prevent transmission over HTTP.

Second biggest web risk: Bad data protection

Mobile app issues: Weak security code

Connection checkers: SSLyze, TestSSL, SSLLabs

Password hunters: GitRob, TruffleHog

Secret keepers: Hashicorp Vault, AWS, Azure, Google tools
